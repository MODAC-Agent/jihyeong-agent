# Week 2: Tech Blog 프로젝트를 개선하며 마주한 이슈 공유

## 개요

Tech Blog 프로젝트는 **AI가 글을 쓰고, 개발자가 방향을 잡는** 한국어 기술 블로그입니다.

지난 week1 마지막 날에 소개했던 이 프로젝트를 꾸준히 개선해오면서 마주쳤던 이슈들을 이번 주에 공유하려 합니다.

Tech Blog 프로젝트는 다음과 같이 3단계에 걸쳐 발전해왔습니다.

1. **Command 단계**: Command를 이용해 프로젝트 글을 작성하던 단계.
2. **Agent 단계**: Agent에 리서치, 작성, 검토 역할을 분담시키던 단계.
3. **Skill 단계**: Skill을 활용해 프로젝트 글을 작성하는 단계. ← **현재**

각 단계를 거치며 느꼈던 문제점, 지금은 어떻게 관리하고 있는지, 그리고 요즘 관심 갖고 지켜보는 지점들을 함께 나눠보겠습니다.

## 1. Command 단계 — 시작은 단순했지만

처음에는 `/write-post`, `/review-post` 같은 Command 하나로 글 한 편을 써내려가는 방식이었습니다.

**느꼈던 문제**

- Command 하나에 리서치·초안 작성·톤 교정까지 전부 몰려 있어, 한 번 실행하면 결과물의 어느 단계가 잘못됐는지 잡아내기 어려웠습니다.
- 프롬프트가 길어질수록 후반부 지시(예: 말투, 인용 규칙)가 무시되는 경향이 있었습니다.
- 글 주제마다 필요한 리서치 깊이가 달랐는데, Command는 이걸 분기 처리하기가 어색했습니다.

---

## 2. Agent 단계 — 역할을 쪼갰더니 생긴 문제

리서치 Agent, 작성 Agent, 검토 Agent로 역할을 분리하면 깔끔해질 줄 알았습니다.

**느꼈던 문제**

- 서브 에이전트는 **독립된 컨텍스트**로 돌기 때문에, 앞 단계의 뉘앙스(예: "이 부분은 가볍게 넘어가자")가 다음 단계로 전달되지 않았습니다.
- 각 Agent에 넘길 입력을 가공하는 비용이 생각보다 컸고, 그 결과 **토큰 소모가 큰 폭으로 증가**했습니다.
- 검토 Agent가 작성 Agent의 의도를 모른 채 피드백을 주다 보니, "틀린 건 아닌데 방향이 안 맞는" 수정 제안이 반복됐습니다.

> week1에서 언급했던 "대부분의 경우 스킬이 더 효율적이다"라는 결론이 여기서 나왔습니다.

---

## 3. Skill 단계 — 지금의 방식

현재는 Skill로 **워크플로우만 정의**하고, 실행은 메인 세션이 그대로 맡도록 바꿨습니다.

- `SKILL.md`에 글 작성 절차(주제 정리 → 아웃라인 → 초안 → 교정)를 체크리스트 형태로 정의.
- 메인 컨텍스트를 공유하므로 앞 단계의 판단이 자연스럽게 이어집니다.
- 서브 에이전트는 **정말로 컨텍스트 오염이 문제가 될 때만** 선택적으로 호출합니다. (예: 대량 레퍼런스 크롤)

---

## 4. 현재 관리 방식

- **글감·초안은 Git으로 관리**: 각 단계 산출물을 커밋으로 남겨, 어디서 방향이 꺾였는지 되짚을 수 있게 했습니다.
- **프롬프트 대신 규칙 문서**: 반복되는 스타일 지시는 `.claude/rules/`로 옮겨 Skill이 자동으로 참조하게 했습니다.
- **세션 마무리는 `session-wrap`**: 글 한 편을 쓰고 나면 어떤 판단이 좋았고 어디서 헤맸는지 분석해 다음 글에 반영합니다.

---

## 5. 요즘 관심 갖고 지켜보는 것들

- **Skill과 Command의 경계**: 어디까지 Skill로 자동화하고, 어디부터 사용자 지시(Command)로 남겨야 할지.
- **긴 글에서의 컨텍스트 관리**: 한 편이 길어질수록 앞부분 톤이 희미해지는 현상을 어떻게 붙잡을지.
- **AI 글의 "개성"**: 일관된 품질을 유지하면서도 기계적으로 읽히지 않게 만드는 장치.
- **독자 피드백 루프**: 발행 후 반응을 다음 글의 Skill 입력으로 다시 흘려 보내는 구조.
