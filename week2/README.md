# Week 2: Tech Blog 프로젝트를 개선하며 마주한 이슈 공유

## 개요

Tech Blog 프로젝트는 **AI가 글을 쓰고, 개발자가 방향을 잡는** 한국어 기술 블로그입니다. Next.js + MDX + velite 기반 정적 블로그로, 개인 프로젝트로 운영 중이에요.

week1 리포트에서 처음 소개했던 이 프로젝트를 꾸준히 개선해오면서 마주쳤던 이슈들을 이번 주에 공유하려 합니다.

Tech Blog 프로젝트는 다음과 같이 3단계에 걸쳐 발전해왔습니다.

1. **v1 · Command 단계**: `/write-post` 하나로 글 한 편을 끝까지 끌고 가던 시기.
2. **v2 · Agent 단계**: `post-writer` / `post-validator` / `post-draft-reviewer` / `post-expression-reviewer` 네 개의 서브 에이전트로 역할을 분담하던 시기.
3. **v3 · Skill 단계**: `blog-*` 스킬 패밀리(현재 10개 + SSOT 하나)로 재구성한 시기. ← **현재**

각 단계를 지나며 어떤 한계에 부딪혀 다음 단계로 넘어갔는지, 지금은 어떻게 관리하고 있는지를 나눠보려 합니다.

## 1. v1 — 한 Command에 모든 걸 담았을 때

`.claude/commands/write-post.md` 하나에 모든 지침을 넣어두고 `/write-post <주제> [URL ...]`로 글을 쓰는 구조였습니다. 자료 수집 우선순위, 복잡도 판단, frontmatter 스키마, MDX 컴포넌트 사용법, 사람처럼 쓰기, 마크다운 함정, 검증 체크리스트, 완료 보고까지 **한 파일 약 460줄**이 한 프롬프트로 올라갑니다.

### 이 단계에서 얻은 것

처음에는 **"AI가 쓴 기술 블로그가 실제로 읽을 만한가"** 라는 프로젝트의 가능성을 검증하는 단계였어요. 커맨드로 찍어낸 글을 직접 읽어봤을 때 "어, 이 정도면 블로그로 올려도 되겠는데?" 싶은 수준이 나왔고, 그 결과를 근거로 프로젝트를 계속 밀고 가기로 결정했죠.

초반의 글 퀄리티 문제나 버그들은 전부 **커맨드 안의 프롬프트(md)를 다듬는 방향**으로 풀어보려 했고, 그렇게 write-post.md는 점점 두꺼워져 갔습니다.

**장점**

- 구조가 단순합니다. 커맨드 파일 하나만 열면 전체 흐름이 눈에 들어와요.
- 실행이 선형적이라 디버깅이 쉬웠습니다.
- 초기 셋업 비용이 거의 없습니다.

### 부딪힌 한계

- **프롬프트 후반부가 묻힙니다.** 커맨드가 400줄을 넘어가면서 마지막 단계인 검증 체크리스트를 지속적으로 놓치는 문제가 생겼어요. 앞쪽 자료 수집 규칙은 잘 지키는데, 뒤쪽의 "em-dash 금지", "콜론 헤딩 금지" 같은 표현 규칙은 반복적으로 어겼습니다.
- **커맨드가 글쓰기 말고는 할 줄 아는 게 없습니다.** 가장 불편했던 지점이에요. `/write-post`는 "글을 작성한다"라는 단일 목적을 가지고 있어서, 작성이 끝난 뒤에 **다듬기**나 **검증만 다시 돌리기** 같은 걸 할 수가 없었습니다. 새 커맨드를 만들까 하다가도 규칙이 write-post.md에 박혀 있어서 복붙 지옥이 예상됐어요.
- **중단 지점이 없습니다.** 기획안만 보고 "이 방향 아니야"라고 끊고 싶어도, 커맨드는 한 번 실행되면 쭉 달려버립니다. 사용자 승인 게이트가 본문 안 지시문으로만 존재하다 보니 일관되게 멈추지 않았어요.
- **수정 단위가 거칩니다.** 검증 규칙 한 줄 바꾸려면 460줄짜리 파일을 건드려야 하고, 옆 규칙을 같이 부수기 쉬웠습니다.

"좀 더 세분화되면 좋겠다"는 생각이 자연스럽게 v2로 이어졌습니다.

## 2. v2 — Agent로 역할을 쪼갰을 때

단일 커맨드가 재사용이 안 된다는 문제를 **역할 분리**로 풀어보려 했습니다. 네 개의 서브 에이전트를 만들었어요.

| Agent                      | 역할                                         |
| -------------------------- | -------------------------------------------- |
| `post-draft-reviewer`      | 기획안 검토 (자료 충분성, 복잡도, 제목 품질) |
| `post-writer`              | 승인된 기획안을 MDX 파일로 저장              |
| `post-validator`           | frontmatter·JSX·내부 링크 등 기계적 정합성   |
| `post-expression-reviewer` | 어조·과장·메타 문장 등 표현 검토             |

처음엔 "역할이 깔끔하게 분리됐으니 잘 굴러가겠지" 싶었습니다. 실제로 몇 가지는 확실히 나아졌어요.

### 해결된 점

- 각 에이전트가 자기 책임에만 집중합니다. writer는 저장까지, validator는 검증까지.
- **글 다듬기·검증 단독 호출이 가능해졌습니다.** v1의 가장 큰 불편이 여기서 풀렸어요. 글 저장 후에 validator만, 또는 expression-reviewer만 다시 부를 수 있게 됐습니다.
- 서브 에이전트는 **독립 컨텍스트**라, writer가 수만 토큰짜리 리서치 결과에 오염되지 않습니다.

### 실제로 써보니 드러난 문제

**1. 규칙이 여러 파일에 흩어졌습니다.**
em-dash 금지, 콜론 헤딩 금지, `**` JSX prop 규칙 등이 writer에도 validator에도 expression-reviewer에도 따로 적혀 있었어요. 규칙 한 줄을 고치려면 기억에 의존하거나 grep으로 관련 파일을 전부 찾아야 했고, 그 과정에서 **누락되는 일도 실제로 생겼습니다**. 한 곳만 고치고 나머지가 뒤쳐지면 에이전트 간에 모순된 규칙이 돌아가게 돼요 (= **SSOT**, Single Source of Truth 위반).

**2. 토큰 비용이 오히려 커졌습니다.**
각 에이전트마다 입력을 가공해서 다시 넘겨야 하고, 공통 규칙도 각자 품고 있으니 총 프롬프트 크기가 단일 커맨드 시절보다 커졌어요.

**3. 실패 처리가 전부 사용자 몫입니다.**
어딘가에서 문제가 생기면 어느 에이전트에서 터진 건지 먼저 찾아야 했습니다. 수정한 뒤에는 그 변경이 다른 에이전트 / 규칙 / 호출 관계에 영향을 주는지 **프로젝트 구성 요소를 일일이 다시 확인**해야 했어요. 자동화된 추적 장치가 없으니 유지보수가 빠르게 피곤해졌습니다.

**4. 단계 간 뉘앙스가 증발합니다.**
**독립 컨텍스트는 장점이자 단점**이에요. 기획 단계의 "이 부분은 가볍게"라는 판단이 writer에게 구조화된 입력으로 떨어지지 않으면 전달되지 않습니다.

**5. "새 글 작성"만 염두에 둔 설계.**
기존 글을 수정하거나 규칙 자체를 편집하는 시나리오는 여전히 공백이었어요.

> **역할 분리 ≠ 컨텍스트 분리**. 이 시점에 분명히 깨달았습니다. 에이전트 경계를 너무 단단히 세우면 오히려 정보가 새나가서, 분리된 각자가 "맞는 일을 다른 맥락에서" 하게 돼요.

## 3. v3 — Skill 패밀리 + SSOT

문제가 명확해졌으니 해결책을 찾아봐야 했어요. 그러다 참고한 게 [gstack](https://github.com/garrytan/gstack)입니다. garrytan 님이 본인 회사와 비슷한 구조로 만든 에이전트 관리 도구인데, 꾸준히 수정·관리되고 있고 에이전트 관리 방법의 선두 주자 중 하나라고 봤거든요.

gstack을 처음 봤을 때 가장 눈에 띈 건 **`agents` 폴더가 아예 없다는 것**이었습니다. 그 자리에 `skills` 폴더가 있고, 몇몇 스킬은 **워크플로우 형태**를 띠고 있었어요. 예를 들어 [autoplan/SKILL.md](https://github.com/garrytan/gstack/blob/main/autoplan/SKILL.md)는 상황에 맞게 워크플로우를 구성하면서 다른 스킬들을 조합해 작업을 진행합니다.

처음엔 의문이 들었습니다. **워크플로우를 통째로 스킬로 만들면 컨텍스트가 과도하게 부풀지 않을까?** 실제로 적용해보니 그런 일은 일어나지 않았어요. 필요한 섹션만 `Read`로 주입하는 방식이라 오히려 가볍게 굴러갑니다.

이 구조를 우리 프로젝트에 맞게 옮긴 결과가 지금의 v3입니다.

### 3-0. 전체 구조

```
.claude/skills/
├── blog-shared/         ← SSOT: 모든 규칙 §RULE-* 로 정의
│   ├── SHARED.md
│   ├── CHANGELOG.md     ← 규칙 수정 이력 자동 기록
│   └── config/domains.md
├── blog-write/          ← 오케스트레이터 (Phase 1~6 + 서브 Phase)
├── blog-research/       ← 자료 수집 (유일하게 sub-agent로 실행)
├── blog-draft-review/   ← 기획안 검토 (GATE 1 이전)
├── blog-writer/         ← MDX 저장
├── blog-validator/      ← 기계 검증 + 자동 수정
├── blog-expression-review/  ← 표현/어조 검토
├── blog-coherence-review/   ← 논리 완결성 통독
├── blog-revise/         ← 기존 글 다듬기 (5가지 패턴)
├── blog-rule-editor/    ← 규칙/스킬 자체 편집 (7개 Rail)
└── blog-banner/         ← 배너 모티프 관리
```

핵심 아이디어는 다섯 가지예요.

### 3-1. SSOT — 규칙은 SHARED.md 한 곳에만

em-dash 금지(`§RULE-EMDASH`), 어조 정의(`§BLOG-VOICE`), frontmatter 스키마(`§FRONTMATTER`) 등 모든 규칙이 `SHARED.md`에 섹션 ID로 정의되어 있습니다. 각 스킬은 필요한 섹션만 `Read`로 주입받아 참조해요.

```
validator → §FRONTMATTER, §RULE-EMDASH, §RULE-COLON-HEADING, ...
writer    → §BLOG-VOICE, §FRONTMATTER, §MDX-*, §RULE-* 전체
```

**효과**: 규칙 한 줄 바꾸면 10개 스킬이 동시에 갱신됩니다. v2에서 "grep으로 찾아다니다 누락"하던 일이 원천적으로 사라져요.

### 3-2. 메인 컨텍스트 공유 + 선택적 sub-agent

v2의 "독립 컨텍스트가 뉘앙스를 잃는다" 문제를 뒤집었습니다. 기본은 **메인 세션에서 실행**이고, sub-agent는 `blog-research`처럼 **대량 원문이 컨텍스트를 오염시킬 때만** 씁니다.

### 3-3. Phase별 명시적 게이트

`blog-write` 오케스트레이터가 Phase 1~6을 순차 실행하되(입력 파싱 + 3.5/4-F/5.5/5.6 서브 Phase 포함), 사용자 개입은 **GATE 1(기획안 승인) 한 곳만 의무**입니다. 나머지 리뷰어들은 블로커를 찾았을 때만 AskUserQuestion으로 개입을 요청해요. v1의 "중단 못 함", v2의 "중단이 애매함" 문제를 한 번에 해결했습니다. - [참고](https://github.com/Zero-1016/tech-blog#%ED%8C%8C%EC%9D%B4%ED%94%84%EB%9D%BC%EC%9D%B8-%ED%9D%90%EB%A6%84)

### 3-4. 실패 학습 루프

writer가 자가 체크리스트를 통과하지 못한 케이스가 `content/tmp/writer-failures.md`에 자동 누적됩니다. 3건 이상 쌓이면 `blog-write` 시작 시 알림이 뜨고, `/blog-rule-editor writer-failures 분석해줘`로 패턴을 뽑아 **규칙 자체를 개선**하는 루프로 이어져요. (주원님 아이디어를 참고하였습니다.)

gstack에는 이 루프를 더 정교하게 만든 **"운영 자기 개선" 섹션**이 각 스킬마다 붙어 있는데, 우리는 아직 거기까진 안 갔고 [CHANGELOG.md](https://github.com/MODAC-Agent/jihyeong-agent/blob/eed0ae709ff9637402db20aa8e6744c69ae38a78/repo/tech-blog/.claude/skills/blog-shared/CHANGELOG.md)로 수정 이력을 쌓으며 한계를 관찰 중이에요. (이 부분은 섹션 6에서 이어 다룹니다.)

최근 1주일(2026-04-12 ~ 04-19) 9건 중 **문제 대응 5건 / 개선 4건**으로 문제 대응이 우세했어요. 구조가 아직 초기라 허점이 계속 드러나는 중이지만, 뒤집어 말하면 writer-failures + CHANGELOG 조합이 **실제로 문제를 드러내는 장치로 작동한다**는 신호이기도 합니다.

### 3-5. 메타 스킬 — `blog-rule-editor`

규칙 수정이 반복되니까, 수정 자체를 스킬로 만들었습니다. 7개의 안전 장치(Rail)를 거쳐요.

1. 사용자 승인 없는 수정 금지
2. 수정 전 `.backups/<파일>-<timestamp>.md` 자동 생성
3. diff 미리보기 + 최종 확인
4. SSOT 위반(같은 규칙 복사) 거부
5. 자기 자신 수정 금지
6. 영향 범위 자동 분석 (grep + 1단계 의존성)
7. 수정 유형별 확인 횟수 조정

규칙 편집뿐 아니라 **새 스킬 추가, writer-failures 분석**도 이 스킬을 경유합니다. v2에서 "수정 후 구성 요소를 일일이 확인해야 했다"는 피곤함이 여기서 상당 부분 해소됐어요.

### 보너스: `blog-revise`로 기존 글도 다루기

v1/v2에서 빠져 있던 "기존 글 수정" 시나리오를 별도 오케스트레이터로 넣었습니다. 5가지 패턴(재검증만 / 부분 수정 / 자료 보강 / 완전 재작성 / 분석만)으로 라우팅하고, 수정 전 백업 + 검증 사이클을 자동으로 돌립니다.

### 3-6. 그런데 blog-write/SKILL.md는 1323줄인데요?

여기서 당연히 의문이 들 수 있어요. **v1의 460줄 커맨드가 컨텍스트 후반부를 놓치던 게 문제라면, 1323줄짜리 `blog-write/SKILL.md`는 왜 안 터지는가?**

핵심 차이는 **"파일 크기"가 아니라 "특정 시점에 컨텍스트에 실제로 로드된 토큰"** 에 있어요. 한 문장으로 답하자면 **컨텍스트가 시간축으로 조각나 있기 때문**이고, 이를 받치는 장치가 네 가지입니다.

**① 스킬은 자동 로드가 아닌 지연 로드**

v1 커맨드는 실행 순간 460줄 전체가 한 프롬프트로 올라가서 글이 끝날 때까지 **내내 살아있었습니다**. 반면 스킬은 트리거될 때만 불러오고, 오케스트레이터 스킬은 Phase별로 하위 스킬을 순차 주입합니다. 예를 들어 `blog-write/SKILL.md`의 Phase 4 지점에는 이런 지시가 적혀 있어요.

```
## Phase 4: writer 호출 (blog-writer SKILL.md 주입)
→ Read .claude/skills/blog-writer/SKILL.md
```

즉 Phase 4에 도달하기 전에는 writer의 상세 규칙(620줄)이 컨텍스트에 없습니다.

**② Phase 간 구조화 변수만 넘김**

각 Phase는 이전 Phase의 원문이 아니라 **요약된 변수**만 넘겨받습니다.

```
topic, user_urls, today
internal_link_candidates     ← Phase 1 결과 (3~5개)
sources_summary              ← Phase 2 결과 (500~1500 토큰)
approved_plan                ← GATE 1 통과 후 기획안
saved_files                  ← Phase 4 저장 경로
validator_report             ← Phase 5 리포트
```

`blog-research`가 수만 토큰짜리 원문을 WebFetch로 읽었어도, 오케스트레이터가 받는 건 500~1500 토큰 요약뿐이에요. 원문은 sub-agent 컨텍스트에서 소진되고 버려집니다.

**③ SHARED.md도 필요 섹션만 주입**

1126줄짜리 `SHARED.md`를 통째로 로드하지 않습니다. writer가 실행될 때는 `§BLOG-VOICE / §FRONTMATTER / §MDX-* / §RULE-*`만, validator는 `§RULE-EMDASH / §RULE-COLON-HEADING / §DOMAIN-WHITELIST`만 Read로 가져가요.

**④ Sub-agent로 컨텍스트 오염 차단**

`blog-research`는 유일하게 **sub-agent 컨텍스트**로 실행됩니다. WebFetch 결과가 오케스트레이터로 흘러들어 오지 않게 일부러 격리해둔 거예요.

정리하면 v1은 "한 번에 460줄 전부 + 본문 전체 + 자료 원문"이 계속 살아있는 **연속 단일 프롬프트**, v3는 "Phase별로 필요한 섹션만 왔다가 사라지는" **조각난 실행 그래프**입니다. 코드 총량은 v3가 훨씬 크지만, **순간 최대 컨텍스트는 오히려 작아요**. 이게 "스킬을 워크플로우로 만들면 컨텍스트가 터지지 않을까?"라는 초기 의문을 실제로 뒤집어준 구조적 답이었습니다.

## 5. 현재 관리 방식

- **규칙 수정은 반드시 `blog-rule-editor` 경유**: `.claude/skills/blog-*` 어떤 파일이든 직접 Edit/Write 금지. 백업/CHANGELOG/영향 범위 분석을 자동으로 돌기 위함.
- **writer-failures 로그가 개선 신호**: 에러 로그가 3건 이상 쌓이면 Layer 1(이번 글 복구) vs Layer 2(규칙 자체 개선) 중에서 후자를 우선 고려합니다.
- **CHANGELOG로 한계 관찰**: 현재는 `blog-shared/CHANGELOG.md`로 규칙 수정 이력을 쌓고, 거기서 "문제 발견으로 수정한 횟수 vs 개선으로 수정한 횟수"의 비율을 관찰합니다. 어느 쪽이 우세한지가 구조의 성숙도를 말해주거든요.
- **자동 백업 + 30일 정리**: `content/posts/.backups/`, `.claude/skills/blog-shared/.backups/` 는 gitignored로 로컬에만 쌓이고, 30일 지난 백업은 수동 정리.
- **세션 마무리는 `session-wrap`**: 글 한 편을 쓰고 나면 "어떤 판단이 좋았고 어디서 헤맸는지"를 분석해 다음 글과 규칙에 반영합니다.

## 6. 마무리 — 세 버전에서 배운 것 + 추후 개선 방향

### 세 버전을 지나며 얻은 세 가지

- **v1 → v2**: 역할을 쪼개는 것만으로는 부족하다. 규칙이 흩어지는 순간 SSOT가 깨지고, 모순된 규칙이 여러 에이전트에서 동시에 돌아간다.
- **v2 → v3**: 컨텍스트 분리는 선택이어야 한다. 기본은 공유, 오염이 예상될 때만 sub-agent로 격리하는 게 맞았다.
- **공통 교훈**: 파일 총량이 아니라 **특정 시점의 순간 컨텍스트**를 설계해야 한다. 1323줄짜리 스킬이 460줄짜리 커맨드보다 가볍게 돌아간 이유가 여기에 있다.

### 다음 단계: 자기 개선 섹션 도입

현재는 CHANGELOG + writer-failures 로그로 한계를 **사람이 관찰**하는 수준입니다. 다음 분기에는 gstack 스타일의 **"운영 자기 개선" 섹션**을 각 스킬에 도입해서, 작업 종료 시 자동으로 학습이 쌓이도록 만들어 볼 예정이에요.

```markdown
운영 자기 개선
마무리하기 전에 이번 내용을 되짚어보세요.

- 예기치 않게 실패한 명령이 있었습니까?
- 접근 방식을 잘못 선택해서 되돌아가야 했나요?
- 프로젝트별로 특이한 점(빌드 순서, 환경 변수, 타이밍, 인증 등)을
  발견하셨나요?
- 누락된 플래그나 설정 때문에 예상보다 시간이 더 오래 걸렸나요?
```

기준은 **"이 학습이 향후 세션에서 5분 이상을 아껴줄까?"**. 이 기준선을 흐리면 노이즈만 쌓인다는 점이 gstack 문서에 명시되어 있어 그대로 참고할 생각입니다.
