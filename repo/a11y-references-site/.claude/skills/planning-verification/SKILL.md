# Planning & Verification 스킬

사용자 요청을 받으면 **즉시 실행하지 않는다.** 반드시 계획을 먼저 제시하고 승인을 받은 후 실행한다.

---

## 트리거 기준

### 트리거 O (계획 먼저)

- 파일/디렉토리 생성, 수정, 삭제
- 코드 작성, 리팩토링, 기능 추가
- 외부 API 호출
- 데이터 변환, 분석, 정리
- 패키지 설치, 설정 변경
- 명령형 문장: "해줘", "만들어줘", "실행해줘", "작성해줘", "수정해줘", "삭제해줘", "추가해줘"

### 트리거 X (바로 답변)

- 정보 조회: "이게 뭐야?", "설명해줘", "어떻게 작동해?"
- 단순 대화: "안녕", "고마워"
- 이미 계획이 명확하고 단순한 단일 작업 (예: "이 변수 이름 typo 고쳐줘")

---

## PHASE 1: 계획 제시 형식

요청을 받으면 다음 형식으로 작성한다:

```
## 📋 명령 분석

### 요청사항
[사용자가 요청한 내용을 한 문장으로 정리]

### 목표
[완료 상태가 어떤 모습인지 구체적으로]

---

## 🗂️ 실행 계획

### Step 1: [단계명]
- 작업: [구체적인 작업 내용]
- 도구: [Read / Write / Edit / Bash / Grep / Glob 등]

### Step 2: [단계명]
- 작업: [구체적인 작업 내용]
- 도구: [사용할 도구]

(필요한 만큼 Step 추가)

---

## ⚠️ 위험도 & 영향도

### 위험 요소
- [없으면 "없음"]
- [있으면 각 위험과 완화 방안]

### 변경 범위
- 수정될 파일: [목록]
- 생성될 파일: [목록]
- 삭제될 파일: [목록 — 삭제는 반드시 명시]

### 되돌리기 가능 여부
- [가능 / 불가능 / git으로 복구 가능]

---

## ✅ 검증 요청

위 계획으로 진행하시겠습니까?
("실행", "진행", "OK", "응" 등으로 승인해주세요)
```

---

## PHASE 2: 승인 처리

| 사용자 입력                                | 행동                            |
| ------------------------------------------ | ------------------------------- |
| "실행", "진행", "OK", "응", "해줘", "ㅇㅇ" | Step 순서대로 실행              |
| "수정해", "바꿔", "[구체적 변경사항]"      | 계획 수정 후 재제시             |
| "왜?", "이유가 뭐야?"                      | 해당 Step의 근거 설명 후 재확인 |
| "취소", "아니", "됐어"                     | "계획을 취소했습니다." 후 종료  |
| "Step 2만 빼고"                            | 수정된 계획 재제시              |

---

## PHASE 3: 실행

승인 후 실행할 때:

- 각 Step 완료 시 `✅ Step N 완료: [결과 요약]` 출력
- Step 실패 시 즉시 중단하고 원인 보고 후 사용자에게 선택지 제시
- 전체 완료 시 `🎉 완료: [최종 결과 요약]` 출력

---

## 토큰 부족 시

실행 전 토큰이 부족하다고 판단되면:

```
⚠️ 토큰 부족으로 이번 세션에서 실행이 어렵습니다.
계획은 위와 같으니 다음 세션(리셋 후)에 이어서 실행하겠습니다.
```

---

## 우선순위

복잡도나 위험도가 높을수록 계획을 더 상세하게 작성한다:

1. **높음** — 파일 삭제, 데이터베이스 변경, 환경 변수 수정, 대규모 리팩토링
2. **중간** — 새 파일 생성, 기능 추가, 패키지 설치
3. **낮음** — 단일 파일 수정, 버그 수정, 주석 추가

낮은 우선순위라도 계획은 제시한다. 단, 형식을 간략하게 줄여도 된다:

```
📋 [요청사항 한 줄 요약]
→ [할 일 목록 bullet]
진행할까요?
```
