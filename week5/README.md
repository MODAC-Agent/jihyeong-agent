# Week 5: AI Agent로 개발 생산성과 워크플로우를 개선한다는 말의 실체

> 처음엔 채팅 한 줄이면 되고, 팀으로 갈수록 “규칙 한 곳”과 “누가 어디서 멈추는지”가 문제가 된다

**이 글을 읽으면 좋은 사람**

- AI에게 코드를 부탁해 본 적은 있지만, **에이전트·스킬·훅** 같은 말이 아직 낯선 사람
- 팀에 Cursor / Claude Code / 비슷한 도구를 **도입하거나 확산**시켜야 하는 사람

---

## 들어가며: “에이전트”를 두 가지로 나눠 생각해 보기

요즘 말하는 **AI 에이전트**를, 이 글에서는 크게 두 층으로 나눕니다.

1. **대화형 보조** — 질문하면 답하고, 파일을 고쳐 달라고 하면 고쳐 준다. 대부분의 사람이 처음 맞닥뜨리는 형태입니다.
2. **업무 절차가 박힌 보조** — 같은 실수를 반복하지 않게 **훅**을 걸고, 같은 설명을 매번 하지 않게 **룰**을 넣고, 여러 단계를 **스킬**로 묶어 둔 형태입니다. 한 사람이 쓰기 좋아지는 단계를 넘어, **팀이 같은 방식으로 일하게** 만드는 쪽에 가깝습니다.

저는 Claude Code·Cursor를 쓰며 디자인 시스템, 접근성 레퍼런스 사이트, 테크 블로그 등에서 이 두 층을 동시에 실험했습니다. 그 과정에서 가장 크게 남은 문장은 이것입니다.

**만들 수 있는 것과, 실제로 쓰이는 것은 다르다.**

기능이 완성됐다고 해서 현장에서 채택되는 것은 아니고, 자동화한 위치가 틀리면 **막히는 사람에게는 아예 닿지 않습니다.** 아래에서는 처음 시작하는 분과 팀에 퍼뜨리는 분이 같이 쓸 수 있게, **짧은 용어 설명 → 개인이 하기 좋은 첫 단계 → 팀 단계에서 깨지는 지점 → 우리가 선택한 워크플로우** 순으로 정리합니다.

---

## 용어 치트시트 (이 글에서만 이렇게 부릅니다)

| 말            | 한 줄 의미                                                      | 비개발 동료에게 비유하면                           |
| ------------- | --------------------------------------------------------------- | -------------------------------------------------- |
| **Hook**      | AI가 파일을 쓰거나 터미널을 쓰기 **전·후**에 돌아가는 자동 검사 | 저장할 때 맞춤법 검사기가 먼저 돌아가는 것과 비슷  |
| **Rule**      | 프로젝트마다 “항상 지켜라”고 AI에게 **미리 주입**하는 규칙 문서 | 팀 스타일 가이드를 옆에 항상 펼쳐 두는 것          |
| **Command**   | `/commit` 같이 **짧은 명령**으로 긴 절차를 부르는 것            | 자주 쓰는 업무를 “매크로”로 묶어 두는 것           |
| **Skill**     | 여러 단계를 **순서와 산출물**까지 정해 둔 재사용 워크플로우     | 체크리스트 + 담당자 순서가 코드로 박힌 업무 절차서 |
| **Sub-agent** | 메인 대화와 **맥락을 분리**한 보조 세션                         | 민감하거나 너무 긴 자료만 따로 방에서 처리하는 것  |
| **SSOT**      | 규칙과 흐름의 **단일 진실 공급원**(한 파일·한 문서가 정본)      | “원본은 이 문서 하나”라고 팀이 합의한 것           |

도구 이름(Claude Code, Cursor 등)은 바뀌어도, **위 개념은 대부분 옮겨 갈 수 있습니다.**

## 1. 처음 하는 사람: 이렇게 시작하면 부담이 적다

### 1-1. 먼저 “반복되는 지시”부터 줄이기

에이전트를 처음 쓸 때는 **거창한 자동화**보다 아래가 효율이 좋습니다.

- **프로젝트 루트에 “이 레포 설명서”를 둔다**  
  예: `CLAUDE.md`에 폴더 구조, 빌드 명령, 컨벤션, 자주 고치는 파일 경로. 매번 같은 설명을 채팅에 붙여 넣지 않아도 됩니다.
- **팀 규칙을 Rule로 옮긴다**  
  “인라인 스타일 금지”, “이 패키지에서만 이렇게 import”처럼 **문장으로 적을 수 있는 것**부터.
- **정말 자주 하는 일만 Command로**  
  커밋 메시지 생성, PR 본문 초안, 릴리즈 노트용 changeset 초안 등 **형식이 비슷한 일**이 좋습니다.

**판단 기준:** “지난주에 같은 설명을 세 번 이상 했나?” → Yes면 Rule이나 Command 후보입니다.

### 1-2. 그다음 “실수 방지” — Hook은 최소한부터

Hook은 강력하지만, 처음부터 많이 걸면 팀이 **왜 막혔는지** 모를 수 있습니다. 추천 순서는 다음과 같습니다.

1. **포맷·린트** — 저장 직후 코드 스타일 맞추기
2. **타입체크 / 테스트** — 작업 끝날 때 한 번
3. **민감 정보·위험 명령** — 팀 정책이 생기면 전역으로

**판단 기준:** “이 실수가 나면 되돌리기 비싼가?” → Yes면 Hook을 검토합니다.

### 1-3. 스킬 vs 서브 에이전트 — 돈(토큰)과 맥락

- **스킬**은 보통 **한 세션 안**에서 단계만 정리하면 됩니다. 팀원에게 설명하기도 쉽습니다.
- **서브 에이전트**는 **긴 원문·민감한 중간 결과**가 메인 대화를 오염시킬 때만 쓰는 편이 낫습니다. 대신 **비용과 전달 설계**가 늘어납니다.

**판단 기준:** “중간 산출물이 메인 채팅에 남으면 다음 지시가 흐려지나?” → Yes면 서브 에이전트 후보입니다.

## 2. 팀에 전파하는 사람: 설득 포인트와 함정

### 2-1. 팀이 “안 쓰는” 이유는 기능 부족이 아닐 때가 많다

디자인 시스템에서 **changeset 자동화 스킬**을 만들었을 때가 그랬습니다. diff 분석, 버전 제안, 릴리즈 노트 초안까지 됐는데, 디자이너분들은 거의 쓰지 않았습니다. 이유는 UI가 어려워서가 아니라,

- 그 도구가 **유료 구독·특정 클라이언트** 위에서만 돌아갔고
- 실제 사용자(디자이너)에게 그 환경이 없었기 때문입니다.

**팀 전파용 한 줄:** “자동화는 **누가 어떤 화면에서** 쓰는지까지 포함해야 완성이다.”

### 2-2. 자동화한 “끝”이 아니라, 사람이 멈추는 “앞”을 보라

디자이너가 막힌 지점은 “changeset 만들어줘” 직전이 아니었습니다. 대략 이렇습니다.

1. Figma에서 토큰을 만져 본다
2. 개발자에게 요청 메시지를 보내려다 **멈춘다**

멈추는 이유는 예를 들면,

- **영향 범위를 모름**
- **내 결정인지 합의 사항인지 애매함**
- **슬랙으로 물어보는 게 더 빠름**

즉 자동화한 것은 **코드 반영 이후**였는데, 막힌 곳은 **의도·영향·합의 이전**이었습니다.

**팀 전파용 한 줄:** “자동화의 **끝부분만** 자동화하면, **시작에서 망설이는 사람**에게는 닿지 않는다.”

### 2-3. 파일이 길다고 실패하는 게 아니다 — “순간 맥락”이 문제다

테크 블로그 파이프라인을 **커맨드 한 덩어리(v1) → 서브 에이전트 분리(v2) → 스킬 패밀리 + SSOT(v3)**로 바꿨습니다.

- v1: 파일이 길어질수록 **프롬프트 뒤쪽 규칙**이 묻힘
- v2: 규칙이 여러 에이전트에 **복제**되면 SSOT가 깨짐
- v3: 규칙은 `SHARED.md` **한 곳**, 스킬은 **필요한 섹션만** 읽고, 단계 사이에는 **요약된 변수만** 넘김

여기서 얻은 교훈을 초심자에게도 이렇게 말할 수 있습니다.

**“파일이 몇 줄이냐”가 아니라, “지금 이 순간 AI 머릿속에 얼마나 많이 올라가 있느냐”가 문제다.**

팀에 설명할 때는 **“규칙은 저장소 한 곳에만 쓴다”**는 약속이 유지보수 비용을 확 줄인다고 말하면 이해가 빠릅니다.

## 3. 우리가 선택한 다음 단계: 디자인 변경 워크플로우 (요약)

위 교훈을 반영해, 디자인 시스템 쪽에서는 **결과물(PR / changeset) 앞단**을 스킬로 쪼개고, 규칙·흐름의 SSOT를 `docs/AGENTS.md`에 두었습니다. 흐름은 다음과 같습니다.

**`/proposal` → `/impact` → `/preview` → `/review` → `/decision`** → (기존) **`/changeset` → `/pr`**

| 단계     | 한 줄                                 | 팀에 말할 때                         |
| -------- | ------------------------------------- | ------------------------------------ |
| proposal | 자연어로 온 요청을 구조화             | “무엇을 바꾸려는지”를 먼저 말로 정리 |
| impact   | 영향 범위를 **읽을 수 있는 요약**으로 | “어디까지 갈지”를 숫자 나열이 아니게 |
| preview  | 보이는 결과로 확인                    | 디자이너는 **그림**이 빠르다         |
| review   | 접근성·개발 비용 등 **다른 관점**     | 합의 전에 한 번 거르기               |
| decision | 진행/보류/반려 + **근거 기록**        | 채팅이 아니라 **남는 기록**          |

문서에 박힌 원칙 중, 팀 전파에 특히 쓸 만한 것만 골라 보면 다음과 같습니다.

- 자연어는 **처음 한 단계에서만** 받고, 이후는 **구조화된 산출물**만 넘긴다
- 단계를 **자동으로 연쇄 실행하지 않는다** — 숨 쉴 틈을 사용자에게 남긴다
- 확인이 필요하면 **선택지 중심 질문**으로 받는다 (비개발자 동료 포함)
- 영향도는 **의사결정에 도움이 되는 요약**이 우선이다

이건 “AI가 똑똑해서”가 아니라 **사람이 어디서 멈추는지**를 줄이기 위한 설계입니다.

## 4. 바로 가져가기: 체크리스트

### 개인이 오늘 할 수 있는 것

- [ ] 이 레포만의 `CLAUDE.md`(또는 동등한 설명서) 초안 1페이지
- [ ] “세 번 이상 반복한 지시” Rule 또는 Command 하나
- [ ] 실수 하나를 Hook 또는 수동 체크로 막기

### 팀에 제안할 때 같이 던질 질문

- [ ] **첫 사용자 1명**은 누구인가 (전사가 아니라 1명)
- [ ] 그 사람은 **어떤 화면·구독**에서 쓰는가
- [ ] 자동화한 단계는 **막히는 지점 앞인가 뒤인가**
- [ ] 규칙의 **정본(SSOT)**은 어디 한 곳인가
- [ ] 실패 사례를 **공유해도 괜찮은가** (학습 속도가 달라짐)

## 5. 아직 열린 질문 (솔직히 적어 둠)

- 단계가 늘면 **슬랙 한 통보다 무거워질 수 있다** — 반드시 1차 사용자와 같이 깎아야 한다
- 영향도 분석이 **너무 많으면** 오히려 결정이 멈춘다
- 미리보기는 **컴포넌트**와 **페이지** 사이에서 비용이 크게 달라진다
- 리뷰는 **자동으로 잡히는 것**과 **사람이 봐야 하는 것**이 섞일 수밖에 없다

완성 선언보다는, **만드는 중에 옆에서 같이 돌려 보고** 고치는 쪽이 현실적입니다.

## 맺으며

AI 에이전트는 처음엔 **채팅을 줄여 주는 도구**처럼 보이다가, 팀으로 갈수록 **규칙을 어디에 두느냐**, **누가 어디서 멈추느냐**, **채택 조건이 갖춰졌느냐**가 문제가 됩니다.

이 글은 그 전환을 한 번 겪은 사람의 스냅샷입니다. 비슷한 실험을 하는 팀이 있다면, [jihyeong-agent](https://github.com/MODAC-Agent/jihyeong-agent)의 week1~week4 기록과 함께 읽어도 좋습니다.
