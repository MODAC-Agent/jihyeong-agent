# AI Agent로 개발 생산성과 워크플로우를 개선한다는 말의 실체

> 처음엔 채팅 한 줄이면 되고, 팀으로 갈수록 “규칙 한 곳”과 “누가 어디서 멈추는지”가 문제가 된다

이 글은 AI에게 코드를 부탁해 본 적은 있지만 에이전트·스킬·훅이 아직 낯선 사람, 그리고 Cursor나 Claude Code 같은 도구를 팀에 도입하거나 퍼뜨려야 하는 사람을 먼저 떠올리고 썼습니다.

---

## 들어가며

요즘 말하는 AI 에이전트를, 여기서는 크게 두 층으로 나눕니다.

**대화형 보조**는 질문하면 답하고, 파일을 고쳐 달라고 하면 고쳐 주는 형태입니다. 대부분이 처음 맞닥뜨리는 층이죠. 그 위에 **업무 절차가 박힌 보조**가 있습니다. 같은 실수를 반복하지 않게 훅을 걸고, 같은 설명을 매번 하지 않게 룰을 넣고, 여러 단계를 스킬로 묶어 두는 쪽입니다. 한 사람이 쓰기 편해지는 수준을 넘어, 팀이 비슷한 방식으로 일하게 만드는 데 가깝습니다.

저는 Claude Code와 Cursor를 쓰며 디자인 시스템, 접근성 레퍼런스 사이트, 테크 블로그에서 이 두 층을 동시에 실험했습니다. 그 과정에서 가장 크게 남은 문장은 하나입니다. **만들 수 있는 것과, 실제로 쓰이는 것은 다르다**는 말입니다. 기능이 완성됐다고 해서 현장에서 채택되는 것은 아니고, 자동화한 위치가 어긋나면 막히는 사람에게는 아예 닿지 않습니다.

아래에서는 잠깐만 쓰는 말을 정리한 뒤, 부담 적은 시작 순서, 팀으로 갈 때 자주 틀어지는 지점, 그리고 디자인 시스템 쪽에서 결과물(PR·changeset) 앞단을 어떻게 나눴는지까지 이어서 적습니다.

## 이 글에서 쓰는 말

| 말            | 한 줄 의미                                                      | 비개발 동료에게 비유하면                                  |
| ------------- | --------------------------------------------------------------- | --------------------------------------------------------- |
| **Hook**      | AI가 파일을 쓰거나 터미널을 쓰기 전·후에 돌아가는 자동 검사     | 저장할 때 맞춤법 검사기가 먼저 도는 것과 비슷             |
| **Rule**      | 프로젝트마다 “항상 지켜라”고 AI에게 미리 주입하는 규칙 문서     | 팀 스타일 가이드를 옆에 항상 펼쳐 두는 것                 |
| **Command**   | `/commit` 같이 짧은 명령으로 긴 절차를 부르는 것                | 자주 쓰는 업무를 매크로로 묶어 둔 것                      |
| **Skill**     | 여러 단계의 순서와 산출물까지 정해 둔, 다시 쓰기 위한 절차 묶음 | 업무 절차서를 실행 가능한 형태로 옮긴 것                  |
| **Sub-agent** | 메인 대화와 맥락을 분리한 보조 세션                             | 너무 긴 자료나 민감한 중간 결과만 따로 방에서 처리하는 것 |
| **SSOT**      | 규칙과 흐름의 단일 진실 공급원, 한 파일·한 문서가 정본          | “원본은 이 문서 하나”라고 팀이 합의한 것                  |

Claude Code나 Cursor 같은 제품 이름은 바뀌어도, 위 개념은 대부분 옮겨 갈 수 있습니다.

## 처음 시작할 때

에이전트를 처음 쓸 때는 거창한 자동화보다, 반복되는 말부터 줄이는 편이 낫습니다. 프로젝트 루트에 이 레포만의 설명서를 둡니다. 예를 들어 `CLAUDE.md`에 폴더 구조, 빌드 명령, 컨벤션, 자주 고치는 파일 경로를 적어 두면 매번 채팅에 같은 설명을 붙여 넣지 않아도 됩니다. 팀에 이미 있는 규칙은 Rule로 옮깁니다. 인라인 스타일 금지, 특정 패키지에서의 import 방식처럼 문장으로 적을 수 있는 것부터면 충분합니다. 정말 자주 하는 일, 커밋 메시지나 PR 본문, 릴리즈 노트용 changeset처럼 형식이 비슷한 일은 Command로 묶기 좋습니다. 지난주에 같은 설명을 여러 번 했다면 Rule이나 Command 후보로 보면 됩니다.

그다음이 실수 방지입니다. Hook은 강력하지만 처음부터 많이 걸면 팀이 왜 막혔는지 설명하기 어려워질 수 있어요. 보통은 저장 직후 포맷·린트, 작업이 끝날 때 타입체크나 테스트 한 번, 팀 정책이 정리되면 민감 정보와 위험한 명령 쪽 순으로 늘리는 식이 부담이 적습니다. 되돌리기 비싼 실수가 있다면 그 지점을 Hook으로 받는지 검토해 보면 됩니다.

스킬과 서브 에이전트는 비용과 맥락의 문제로 고르는 편이 낫습니다. 스킬은 한 세션 안에서 단계만 정리하면 되는 경우가 많고, 팀원에게 설명하기도 쉽습니다. 서브 에이전트는 긴 원문이나 민감한 중간 결과가 메인 대화를 오염시킬 때 쓰는 편이 낫고, 대신 토큰 비용과 전달 설계가 늘어납니다. 중간 산출물이 메인 채팅에 남으면 다음 지시가 흐려진다면 서브 에이전트를 떠올리면 됩니다.

## 팀에 퍼뜨릴 때

팀이 안 쓰는 이유가 항상 기능 부족은 아닙니다. 디자인 시스템에서 changeset 자동화 스킬을 만들었을 때가 그랬어요. diff 분석, 버전 제안, 릴리즈 노트 초안까지 됐는데 디자이너분들은 거의 쓰지 않았습니다. UI가 어려워서가 아니라, 그 도구가 유료 구독과 특정 클라이언트 위에서만 돌아갔고 실제 사용자에게 그 환경이 없었기 때문이었습니다. 자동화는 누가 어떤 화면에서 쓰는지까지 포함해야 비로소 완성에 가깝다는 걸 다시 배웠습니다.

또 하나는 자동화한 “끝”이 아니라 사람이 멈추는 “앞”을 보는 일입니다. 디자이너가 막힌 지점은 “changeset 만들어줘” 직전이 아니었습니다. Figma에서 토큰을 만져 본 뒤, 개발자에게 요청 메시지를 보내려다 멈추는 그 앞이었어요. 영향 범위를 모르거나, 이게 내 결정인지 합의 사항인지 애매하거나, 슬랙으로 물어보는 게 더 빠른 경우였습니다. 자동화한 것은 코드 반영 이후였는데, 막힌 곳은 의도·영향·합의 이전이었습니다. 자동화의 끝부분만 자동화하면, 시작에서 망설이는 사람에게는 닿지 않는다는 말이 바로 이 상황입니다.

테크 블로그 파이프라인은 커맨드 한 덩어리였던 시기, 서브 에이전트로 역할을 나눴던 시기, 스킬 패밀리와 SSOT로 묶은 시기를 거치며 모양이 바뀌었습니다. 파일이 길어질수록 프롬프트 뒤쪽 규칙이 묻히는 문제가 있었고, 역할만 나누면 규칙이 여러 곳에 복제되어 SSOT가 깨지기도 했습니다. 나중에는 규칙을 `SHARED.md` 한 곳에 두고, 스킬은 필요한 섹션만 읽게 하며, 단계 사이에는 요약된 변수만 넘기는 쪽으로 정리했습니다. 여기서 얻은 말은 단순합니다. 파일이 몇 줄이냐가 아니라, 지금 이 순간 AI에게 실제로 올라가 있는 맥락이 얼마나 무거우냐가 문제라는 것입니다. 팀에 설명할 때는 규칙을 저장소 한 곳에만 쓴다는 약속이 유지보수 비용을 크게 줄인다고 말하면 이해가 빠릅니다.

## 디자인 변경을 앞단부터 나눈 이유

위 교훈을 반영해, 디자인 시스템 쪽에서는 PR과 changeset보다 앞을 스킬로 쪼개고, 규칙과 흐름의 SSOT를 `docs/AGENTS.md`에 두었습니다. 흐름은 `/proposal`에서 `/impact`, `/preview`, `/review`, `/decision`을 거친 뒤, 기존에 있던 `/changeset`과 `/pr`로 이어집니다.

proposal은 자연어로 온 요청을 구조화합니다. 무엇을 바꾸려는지를 먼저 말로 정리하는 단계죠. impact는 영향 범위를 사람이 읽을 수 있는 요약으로 만듭니다. import가 몇 군데인지 같은 숫자 나열보다, 어디까지 갈지 판단에 도움이 되게요. preview는 바뀐 모습을 보이는 쪽으로 확인합니다. 디자이너에게는 그림이 숫자보다 빠른 경우가 많습니다. review는 접근성이나 개발 비용 같은 다른 관점을 한 번 거칩니다. decision은 진행·보류·반려와 근거를 남깁니다. 채팅 한 줄로 끝나면 사라지는 말이 아니라, 나중에 다시 찾을 수 있는 기록으로요.

문서에 박힌 원칙도 같은 방향입니다. 자연어는 처음 한 단계에서만 받고 이후는 구조화된 산출물만 넘깁니다. 단계를 자동으로 연쇄 실행하지 않고, 사용자에게 숨 쉴 틈을 남깁니다. 확인이 필요하면 선택지 중심으로 묻습니다. 비개발자 동료도 응답하기 쉽게요. 영향도는 의사결정에 도움이 되는 요약이 우선입니다. 이건 AI가 특별해서가 아니라, 사람이 어디서 멈추는지를 줄이기 위한 설계입니다.

---

## 맺으며

AI 에이전트는 처음에는 채팅을 줄여 주는 도구처럼 보이다가, 팀으로 갈수록 규칙을 어디에 둘지, 누가 어디서 멈추는지, 도구를 쓰려면 어떤 환경이 갖춰져야 하는지가 문제가 됩니다. 단계를 늘리면 슬랙 한 통보다 무거워질 수도 있고, 영향도를 너무 많이 보여 주면 오히려 결정이 멈출 수도 있습니다. 미리보기는 컴포넌트와 페이지 사이에서 비용이 크게 달라지고, 리뷰는 자동으로 잡히는 부분과 사람이 봐야 하는 부분이 섞일 수밖에 없습니다. 그래도 완성을 선언하기보다는, 만드는 중에 1차 사용자 옆에서 같이 돌려 보고 고치는 쪽이 현실적이라고 느낍니다.

이 글은 그 전환을 한 번 겪은 사람의 스냅샷입니다. 비슷한 실험을 하는 팀이 있다면 [MODAC-Agent/jihyeong-agent](https://github.com/MODAC-Agent/jihyeong-agent)의 week1부터 week4 기록과, DDS 쪽의 `docs/AGENTS.md`와 함께 읽어도 좋습니다.
